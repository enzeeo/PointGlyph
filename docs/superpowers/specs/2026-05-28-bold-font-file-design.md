## Problem Statement

Add support for bold fonts in PointGlyph without adding synthetic bolding or a
new font-weight system.

The intended UX is:

```bash
pointglyph "CONVERG3D" --font ./fonts/Inter-Regular.ttf --output ./exports/normal
pointglyph "CONVERG3D" --font ./fonts/Inter-Bold.ttf --output ./exports/bold
```

## Current Behavior

`pointglyph` already accepts any existing `.ttf` or `.otf` path through `--font`.
Because `render_text_mask()` uses Pillow's `ImageFont.truetype()`, passing a real
bold font file already produces bold glyph geometry.

The missing work is to make this behavior explicit, tested, and documented.

## Premises

1. "Bold" means a real bold font file, for example `Inter-Bold.ttf`.
2. The CLI should stay simple: `--font` remains the only font selector.
3. PointGlyph should not add a `--bold` flag in this pass.
4. PointGlyph should not resolve system font families or infer font weights.
5. The manifest should continue recording the actual font basename so consumers
   can see which file produced the asset.

## Approaches Considered

### Approach A: Real Bold Font File

Use an actual bold `.ttf` or `.otf` through the existing `--font` path.

Effort: S
Risk: Low

Pros:
- Produces true font-designed bold glyphs.
- Keeps CLI and rendering pipeline unchanged.
- Avoids distorted synthetic thickening.

Cons:
- User must have the bold font file.
- Tests need a portable way to find or skip a bold system font.

Reuses:
- `src/pointglyph/cli.py`
- `src/pointglyph/text_mask.py`
- `tests/conftest.py`
- `tests/test_cli.py`
- `README.md`

### Approach B: Synthetic `--bold`

Add a CLI flag that thickens text using Pillow stroke or offset drawing.

Effort: M
Risk: Med

Pros:
- Easier one-flag UX.
- Works even when the user only has a regular font file.

Cons:
- Not true typographic bold.
- Can distort counters, spacing, and tight glyphs.
- Adds rendering behavior users may mistake for font weight support.

### Approach C: Font Weight Metadata Only

Add manifest metadata such as `"fontWeight": "bold"` without changing rendering.

Effort: S
Risk: Low

Pros:
- Useful to downstream consumers.
- Minimal implementation cost.

Cons:
- Does not make output bold.
- Needs either user input or unreliable filename inference.

## Recommended Approach

Use Approach A. It matches PointGlyph's asset-generator model: exact font file in,
exact rendered particle geometry out.

## Implementation Plan

1. Add a bold font fixture.
   - Extend `tests/conftest.py` with `BOLD_FONT_CANDIDATES`.
   - Include common macOS/Linux candidates, for example Arial Bold and DejaVu Sans
     Bold.
   - Skip bold-specific tests when no supported bold font exists.

2. Add bold rendering coverage.
   - Add `test_render_text_mask_accepts_bold_font`.
   - Assert mask content exists and bounds are nonzero.

3. Add CLI/manifest coverage.
   - Add a CLI test that runs with `bold_font_path`.
   - Assert required files are generated.
   - Assert `manifest["font"]` equals the bold font basename.

4. Document bold usage.
   - Add a README section showing normal and bold font-file examples.
   - State that PointGlyph uses the exact supplied font file.
   - State that synthetic `--bold` is intentionally not included.

5. Verify.
   - Run `uv run pytest -q`.
   - Run one smoke command with a real bold font path if available.

## Success Criteria

- A user can generate bold particle assets by passing a real bold font file.
- The behavior is documented in README.
- Tests prove bold font files render through both the mask layer and CLI.
- No `--bold` flag, font-family lookup, or synthetic thickening is added.
